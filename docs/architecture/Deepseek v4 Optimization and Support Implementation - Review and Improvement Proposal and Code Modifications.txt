# Deepseek v4 Optimization and Support Implementation - Review and Improvement Proposal

I've reviewed the codebase's DeepSeek V4 customizations against the architecture plan (`docs/architecture/deepseek-v4-optimization-plan.md`). The implementation is ambitious and touches many parts of the pipeline. Here's my analysis, key improvement areas, and potential pitfalls.

---

## Key Improvement Areas & Fix Proposals

### 1. Interleaved Thinking – Reasoning Content Handling Might Not Be Fully Aligned with V4

**Plan requirement**:  
When tools are present, V4 preserves **all** reasoning content across **all** turns, including across user message boundaries. When no tools are present, the default behaviour drops reasoning from earlier turns (except the latest).

**Current code** (inferred from context and provider adapter logic):  
The code still contains a `filterDeepSeekReasoningMessages()` function (likely in `provider.go`). The V4‑specific logic should differentiate based on whether the system message contains tool definitions for that conversation. The plan specifies that when tools are present, we must **not strip** `reasoning_content` from any turn, and we should preserve the full interleaved thinking chain.

**Proposal**:  
- In `BuildMessagesFromPrompt`, explicitly track whether tool definitions will be sent to the model (i.e., the conversation has tools).  
- Pass a flag to the provider adapter to indicate that the message list must retain all reasoning content.  
- Remove or conditionally disable the old `filterDeepSeekReasoningMessages` for V4 when tools are active.  
- Add a test that simulates a multi‑turn tool‑using conversation and verifies reasoning is preserved.

---

### 2. Cache‑Aware Prompt Construction – Prefix Hash Checkpoint Is Not Persisted Across Calls

The plan expects that when the stable prefix changes (cache break), PicoClaw logs it and tracks the new hash. The code in `BuildMessagesFromPrompt` computes `prefixHash` and logs a break if `lastPrefixHash` changed, but `lastPrefixHash` is stored on the `ContextBuilder` which is **per‑agent, per‑request** — it is not persisted across agent restarts.

**Issue**:  
The prefix hash cache diagnostic is ephemeral; after a restart, the first call will not log a break even if the workspace content changed because `lastPrefixHash` resets to 0.

**Proposal**:  
Store the last known prefix hash in a durable session metadata or a small persistent file alongside the context builder, so that cross‑restart cache behaviour can be diagnosed and recorded. For now, this is low priority, but for production monitoring of caching efficiency it would be useful.

---

### 3. Adaptive Compression – Summarization Trigger May Still Fire Too Early for 1M‑token Models

The plan’s `maybeSummarize` uses the legacy message‑count and token‑percent thresholds even after the refactoring. Although the `compressionStrategy` scales thresholds for `adaptive` and `conservative`, the per‑message token estimate is still based on the `chars * 2 / 5` heuristic, which is inaccurate for DeepSeek’s tokenizer (the plan itself notes a 20‑40% error rate).

**Improvement**:  
- For V4 models, always prefer API‑reported usage (from streaming `include_usage` or response `usage` block) to measure real token consumption.  
- Use the actual token count reported in `exec.response.Usage` to determine whether to summarise, rather than relying solely on the heuristic. This can be done after the LLM call, adjusting the next turn’s thresholds.

---

### 4. Dynamic Thinking Mode – Only Coarse‑Grained Control

The current implementation switches thinking **off** after a tool execution and **on** for fresh or steering‑resumed iterations. This is a good start, but the plan mentions more nuanced strategies, e.g., switching to `think‑max` for complex reasoning tasks and `think‑high` for general tasks.

**Proposal**:  
Enhance `resolveThinkingLevelForIteration` with a complexity score (reuse the model router’s complexity features) to decide whether to use `think‑high` or `think‑max` instead of always reverting to the configured level. This would give a three‑tier dynamic thinking: non‑think (post‑tool), think‑high (normal), think‑max (high‑complexity initial reasoning).

---

### 5. Streaming Integration – Error Handling on Prefix Completion Edge Cases

The streaming path in `callWithStreaming` does not seem to handle the case where the provider streams but the connection is interrupted before the final `include_usage` chunk. In such cases, the usage data might be absent, and subsequent context budget calculations would fall back to the heuristic, which may be inaccurate.

**Improvement**:  
- Ensure that if streaming fails after some chunks, the partial response is still usable and that a fallback usage estimate is logged.  
- If the `onChunk` callback accumulates enough text, consider using it for the final response when the stream is broken but sufficient content was received.

---

### 6. DSML Tool‑Call Parser – Missing Integration Points

The plan expects a `dsml_parser.go` that can parse `<|DSML|tool_calls>…` markup from local model outputs. The code snippets do not show its full integration; the `openai_compat` provider should check if the response text contains DSML when `tool_calls` array is empty. Without this, local DeepSeek V4 models using the raw API (e.g., vLLM) would fail to execute tools.

**Proposal**:  
- Ensure `provider.go`’s response handling looks for DSML markers and calls `ParseDSMLToolCalls`.  
- Add integration tests with a mock that returns DSML‑formatted text and verifies tool calls are extracted.

---

### 7. Strict Mode for Tool Calls – Schema Validation Before Sending

The plan mentions a schema validation helper that checks function definitions for strict‑mode compliance before sending them to DeepSeek. I can’t see this validation in the provided code (though it may exist in a separate file). If it’s missing, sending non‑compliant schemas will result in 400 errors.

**Improvement**:  
- Implement a validator that scans all registered tool schemas and ensures every object has `additionalProperties: false` and all properties are in `required`.  
- If the validator fails, either skip strict mode for that request or log a warning and remove the offending tool from the request (which might be worse).  
- Consider a `strict_tool_calls_strict_schema` option that forces schema compliance and rejects startup if any tools cannot be made strict.

---

### 8. JSON Output Mode – May Cause Issues When Used Without `response_format`

The code sets `response_format` in `llmOpts` and calls `ensureJSONInstructions`, but the DeepSeek API requires that `response_format` is sent as a top‑level parameter, not inside `llmOpts`? Actually, the DeepSeek V4 API uses `response_format` at the root level of the request body (like OpenAI). The `llmOpts` map is presumably passed to the provider, and the provider should extract it. I assume the provider adapter handles this correctly. However, if the provider is not DeepSeek, the `response_format` might be ignored.

**Potential issue**: The JSON instructions are appended to the system message only if not already present; but they might need to be in the **user** message for DeepSeek (the API docs state that when `response_format` is `json_object`, a system or user message must explicitly mention JSON format). The code currently appends the instruction to the system message. That should work.

---

### 9. Context Partitioning – No Enforcement on I/O of Partition Budgets

`isOverPartitionBudget` is defined but I don’t see it called in the proactive budget check before LLM call (`isOverContextBudget` in `pipeline_setup.go` still uses the flat check). The partition‑based overflow detection is not yet integrated into the turn lifecycle.

**Proposal**:  
- Replace or augment `isOverContextBudget` in `SetupTurn` with `isOverPartitionBudget` when a partition config is active.  
- If the history partition overflows, trigger targeted compression of the oldest turns rather than the generic 50% drop.  
- If the injected context partition overflows, truncate the injected items before building messages.

---

### 10. Token Estimator Still Heuristic – Cache Efficiency Could Be Better with Real Tokenizer

The plan acknowledges that DeepSeek’s tokenizer differs and the heuristic can cause mis‑estimation. While the proactive + reactive design mitigates most problems, using the API’s reported usage for future turns would improve accuracy. The `SessionCostTracker` already records usage, but that data is not fed back into the context budget for the same session’s subsequent turns.

**Suggestion**:  
- Maintain a rolling estimate of tokens‑per‑character per model in a session, and use that weighted average for future `EstimateMessageTokens` calls when API usage data isn’t available yet.

---

## Overall Assessment

The modifications align well with the DeepSeek V4 optimization plan, covering caching, thinking control, streaming, context partitioning, and cost tracking. The codebase is structured enough that many features are pluggable and testable.

**Key Risks**:
- The interleaved thinking (reasoning content) handling may not fully support V4’s tool‑call mode as documented, potentially breaking tool‑chain continuity.  
- The partition‑based compression is not yet wired into the actual turn checklist; the proactive check still uses the flat budget.  
- The DSML parser integration is absent in the visible code, possibly limiting local deployments.  
- Strict mode missing schema validation could cause silent failures.

I recommend prioritising the **interleaved thinking fix** and **partition enforcement** before production, as those directly impact conversation quality and context management efficiency.

## 4. Strict Mode – Schema Validation Before Sending

**Current issue**: The plan calls for validating tool schemas for strict-mode compliance; this is missing from the visible code.

**Solution**: Add a schema validator and either skip strict mode or drop non-compliant tools.

### File: `pkg/providers/openai_compat/schema_validator.go` (new file)

Create a simple validator:

```go
package openai_compat

import (
    "fmt"
    "github.com/sipeed/picoclaw/pkg/providers"
)

func ValidateStrictToolSchemas(tools []providers.ToolDefinition) ([]providers.ToolDefinition, error) {
    var clean []providers.ToolDefinition
    for _, t := range tools {
        if t.Function.Parameters == nil {
            continue
        }
        schema, ok := t.Function.Parameters.(map[string]any)
        if !ok {
            continue
        }
        if err := enforceStrictSchema(schema, ""); err != nil {
            return nil, fmt.Errorf("tool %q: %w", t.Function.Name, err)
        }
        clean = append(clean, t)
    }
    return clean, nil
}

func enforceStrictSchema(schema map[string]any, path string) error {
    if t, _ := schema["type"].(string); t == "object" {
        props, ok := schema["properties"].(map[string]any)
        if ok && len(props) > 0 {
            if _, has := schema["additionalProperties"]; !has {
                schema["additionalProperties"] = false
            } else if ap, ok := schema["additionalProperties"].(bool); ok && ap {
                return fmt.Errorf("path %s: additionalProperties must be false for strict mode", path)
            }
            required, _ := schema["required"].([]any)
            for name := range props {
                if !containsAny(required, name) {
                    // Log warning, but automatically add to required if missing
                    // (automatic fix rather than error)
                    schema["required"] = append(required, name)
                }
            }
        }
        for k, v := range props {
            if sub, ok := v.(map[string]any); ok {
                if err := enforceStrictSchema(sub, path+"."+k); err != nil {
                    return err
                }
            }
        }
    }
    return nil
}
```

## 5. Streaming Error Handling – Graceful Fallback

**Current issue**: If streaming fails after partial chunks, the response is lost and the error propagates.

**Solution**: In `callWithStreaming`, accumulate partial content and return a best-effort response on error, with usage estimation.

### File: `pkg/agent/pipeline_llm.go`, in `callWithStreaming`

Modify the function to capture accumulated content and fall back to it if the stream fails:

```go
var accumulatedText string
onChunk := func(text string) {
    accumulatedText = text
    // ... event emission as before
}

resp, err := sp.ChatStream(ctx, messages, tools, model, opts, onChunk)
if err != nil {
    if accumulatedText != "" {
        // Return partial response as a fallback
        logger.WarnCF("agent", "Stream failed, returning partial response",
            map[string]any{"error": err.Error(), "partial_len": len(accumulatedText)})
        return &providers.LLMResponse{
            Content:      accumulatedText,
            FinishReason: "interrupted",
            Usage: &providers.UsageInfo{
                // Heuristic estimate for partial stream
                PromptTokens: EstimateMessageTokens(combinedMsg),
                CompletionTokens: tokenizer.EstimateMessageTokens(providers.Message{Content: accumulatedText}),
            },
        }, nil
    }
    return nil, err
}
```

This requires a `combinedMsg` for context; you can compute it from the messages slice.
